# 第16章「［離陸編］要件定義の仕上げ」感想文
1. この章の要点（自分の言葉で3行）：ここまでに、「UI」「機能」「データ」についてまとまり、基本的にはこれで要件定義が終わったと言ってもよいところまできたが、これだけだと、ワークセットが漏れていたり、行動シナリオ自体が足りなかったりしたとき、今の時点の要件だけでソフトウェアを完成させて実際に使い始めて、そのときになってその「不足」に気付くことになるため、「要件定義が不十分だった」とあとで言うはめになることが多い。そのような事態を事前に防止するために、CRUDマトリックス（Create:生成・Reference:参照・Update:更新・Drop/Delete:消去）という表を作り検証する。エンティティの1つずつに対して、C/R/U/Dがあるかを確かめ、どれか1つでもない場合は、ワークセットと行動シナリオの追加の作業を繰り返し精度を高め、最後にプロジェクトの必要性に応じて、応答性や可溶性などの各種の非機能要件を定義していく。
2. なるほど／腑に落ちた点（1〜2個）：ここまでは自分で作ってきたものであって、全てが完璧に出来上がっているわけではないから、それをどこにも落ち度がない、確かなものにするために検証しなければならない。
3. 身近な例で言いかえると（教本の例、または「入金消込」のような“想像でわかる”例でOK。実際の自社業務は知らなくてよい）：厨房でいうと、メニュー表（UI）・調理の段取り（機能）・食材の置き場（データ）がそろい、開店できそうに見える段階になっても、実際に店を開けると、「あの食材、補充も廃棄も段取りに入っていなかった」と営業中に気付くようなもの。そうならないために、開店前に冷蔵庫を棚卸しして、食材ごとの出し入れの抜けを一覧で点検しておく。食材にやることは、入れる・見る・詰め替える・捨てるの4つだけ。仕入れて終わりでは店は回らず、使い切って空にするまでが一巡。棚卸し表に段取りをたどって印をつけていき、印のつかないマスは、その食材のその作業が段取りから抜けている合図になる。抜けを見つけたときは、仕入れ・提供の段取りを足し、全部の印がつくまでやり直す。棚卸しが済んだら、最後に「提供の速さ」「休まず店を開けられるか」など店の質を詰める。つまり、出し忘れ・捨て忘れをあぶり出す点検表である冷蔵庫の棚卸し表を使い、不備がないよう開店前に穴をふさぐ。
4. 疑問・もっと知りたい点（1個）：検証で抜けやバグがあると分かり、それまでの設計を見直すとき、どんな修正方法だったら、確実に間違いない設計に作り直すことができるのか
5. 前の章・V字（開発工程）とのつながり（1行）：受入テストにつながる、要件定義の仕上げ

①作って満足せず“検証”すると、なぜ事故が減る？
- 作るだけでは自分が想定したものを形にしただけで、抜けやバグなど、想定そのものが正しい保証はなく、たまたま動いたものを、確実に動くものにするために検証が必要になる。
