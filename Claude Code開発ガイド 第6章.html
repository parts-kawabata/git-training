Prologue ／ あらすじ

“良いコード”という、優等生の罠。

“

田中のリファクタリングの成功は、いつのまにか社内で評判になっていた。気づけば他のメンバーも次々とClaude Codeを使いはじめ、AIが書いたコードのプルリクエストが、田中の手元に日に日に積み上がっていく。

その一枚をめくった指が、ふと止まる。「——今の要件に対して、これは明らかに過剰だ」。必要以上に込み入った構造。丁寧すぎる抽象化。それは、“良いコード”を書こうとするAIならではの、いわば優等生の癖のようなものだった。一件ずつ、丁寧に。そう思って読み進めるほどに、時間だけが静かに溶けていく。

行き詰まった田中は、また佐藤さんのデスクを訪ねた。事情を聞いた佐藤さんは、こともなげに言う。「AIが書いたコードは、AIにレビューさせればいい」。

その一言を境に、田中は“一人で抱える開発”から、仕組みで回す「ワークフロー」の設計へと足を踏み入れていく。——画面の向こうで、AIはどこまで開発の相棒になれるのか。そして、チームのための“仕組み”とは、いったいどんな形をしているのか。

What happened ／ 舞台裏

田中さんが Claude Code を使って行っていたこと

社内で注目を集めた田中さんの周りでは、いつしか他のメンバーもClaude Codeを使いはじめ、AIが書いたコードのPRが日に日に増えていった。今度はそれを“さばく”ための工夫を、順を追って覗いてみよう。
Claude Code とは ── 指示を日本語で書くだけで、コードのレビューや生成、外部ツール連携までこなしてくれる「AIの開発パートナー」のこと。

AIレビュー

例：佐藤さんがClaude Codeにレビューを依頼した際のプロンプト

田中さん → Claude Code
このPRの差分をレビューしてください。以下の観点で問題点を指摘してください：

  1. セキュリティリスク（SQLインジェクション、XSS、認証バイパス等）
  2. 不必要な複雑さ（現在の要件に対してオーバーエンジニアリングな設計）
  3. テスト漏れ（テストされていないエッジケース）
  4. CLAUDE.mdの規約違反
  5. パフォーマンスの問題（N+1問題、不要なDBアクセス等）

  問題を見つけたら、該当コードと改善案をセットで提示してください。

佐藤さんのアドバイスに従ってAIレビューを試してみた結果、9件のPRのうち、機械的なチェックはClaude Codeが処理し、田中さんは設計判断に集中でき、レビュー時間は半分以下になった。

🔰 はじめての人へ ─ ここに出てくる言葉
  • PR（プルリクエスト）：自分の書いたコードをチームに取り込んでもらうための“変更の申請”。
  • SQLインジェクション／XSS／認証バイパス：いずれもよくある攻撃の入口。放っておくと情報漏えいや乗っ取りにつながる。
  • オーバーエンジニアリング：今の要件に対して、つくり込みすぎ・複雑にしすぎている状態。
  • N+1問題：1回で済むはずのデータベース問い合わせを、何回も繰り返してしまう遅さの原因。
  • CLAUDE.md：チームの規約や方針を書いておく“AIへのルールブック”。

hooksの設定の追加

佐藤さんの例

.claude/settings.json
{
    "hooks": {
      "PreToolUse": [
        {
          "matcher": "Bash",
          "hooks": [
            {
              "type": "command",
              "command": ".claude/hooks/pre-commit-check.sh"
            }
          ]
        }
      ],
      "PostToolUse": [
        {
          "matcher" "Write|Edit",
          "hooks": [
            {
              "type": "command",
              "command": ".claude/hooks/format-java.sh"
            }
          ]
        }
      ]
    }
}
🧩 設定のポイント
  • 設定はJSON形式で書く
  • まず前提として、シェルコマンドを実行するときは Bash ツール、ファイルを新規作成するときは Write ツール、既存ファイルを編集するときは Edit ツール、ファイルを読むときは Read ツールというように、Claude Codeは内部で「ツール」を使い分けている
  • PreToolUse は、そのツールを実行する前に走るフックで、matcher に Bash を指定しているから、Claude Codeがシェルコマンド（git commit でもビルドコマンドでも何でも）を実行しようとしたタイミングで、事前にCheckstyleのチェックとテスト実行が自動で走り、チェックが通らなかったら、コマンドの実行がブロックされる
  • PostToolUse は、ツールを使った後に走るフックで、matcher に Write|Edit を指定しているから、ファイルの新規作成や編集のたびにgoogle-java-formatによる自動フォーマットがかかり、AIが書いたコードも常にチームのフォーマットに統一される
  • hooksはプッシュする前の、もっと手前の段階でチェックをかける仕組み
  • Gitで管理できるため、チームで共有すれば全員のClaude Codeに同じ品質ゲートがかかる
🔰 ポイント
  • hooks（フック）：Claude Codeがツールを使う前後に、自分で決めた処理を自動で割り込ませる仕組み。
  • Checkstyle / google-java-format：Javaのコードがルールどおりか検査したり、見た目を自動で整えたりする道具。
  • 品質ゲート：「ここを通らないと先に進めない」という自動の関所。

意図的にCheckstyle違反のあるコードをClaude Codeに生成させ、コミット

Claude Codeがgit commitコマンドを実行しようとした瞬間、PreToolUse フックが発火してCheckstyleが違反を検出し、コマンドの実行がブロックされた

🔰 いま起きたこと
  • コミット：変更を記録として確定する操作。ここで関所（hooks）が働き、違反があると確定そのものが止まる。

MCPサーバーの登録と連携

claude mcp add jira-server -- npx -y @anthropic/mcp-jira

claude mcp add の後にサーバーの名前をつけて、-- の後に起動コマンドを書くだけで登録完了。

田中さん → Claude Code
jiraチケット　TRACK-142 の内容を確認して、このチケットに対応する
  PRの説明文を作成してください。

Claude CodeがjiraのMCPサーバーを通じてチケット情報を取得し、PRの説明文を自動生成した。チケットのタイトル、概要、受け入れ条件が反映された説明文。これまでブラウザでjiraを開き、内容を読み、手動でPRの説明欄に転記していた作業が、一つのプロンプトで完了した。

🔰 はじめての人へ
  • MCPサーバー：Claude Codeを、jiraやGitHubなど外部ツールとつなぐ“接続口”。
  • jira：タスクや課題（チケット）を管理する、開発現場の定番ツール。
  • npx：必要なツールをその場で取り寄せて実行できる、Node.jsの便利コマンド。

田中さんは生成されたドキュメントを読み、エンドポイントの一覧は正確で、リクエスト・レスポンスの型も合っていたが、いくつか手直しが必要だった。

ドキュメント更新用のコマンドの生成

.claude/commands/update-docs.md
<!-- .claude/commands/update-docs.md -->
  $ARGUMENTS のモジュールについて、以下のドキュメントを最新のコードに基づいて更新してください:

  1. README.md - モジュール概要、API一覧、設定方法、依存関係
  2. API仕様書 - エンドポイント、パラメータ、レスポンス、使用例

  既存のドキュメントがあればその構成を維持しつつ内容を更新し、なければ新規作成してください。

これで /update-docs timetracker-notification と打つだけで、指定モジュールのドキュメントが最新化され、毎回プロンプトを書き直す必要がなくなった。

  • カスタムスラッシュコマンド：よく使う指示を /名前 の一語で呼び出せるよう、あらかじめ登録しておいたもの。

Column ／ コラム

AI駆動開発のワークフロー全体像

CLAUDE.md、Skills、Plan mode、Subagent、hooks、MCP、ドキュメント生成。これまで章をまたいで登場した“7つのピース”は、組み合わさってはじめて本当の力を発揮する。

1

CLAUDE.mdでルールを定義する（第2章）
チームのコーディング規約、アーキテクチャ方針、禁止事項をCLAUDE.mdに記載する。Claude Codeはセッション開始時にこれを読み込み、ルールに沿ったコードを生成する。

2

Skillsで作業を定型化する（第2章）
レビュー、テスト生成、リファクタリングなどの定型作業を、カスタムスラッシュコマンドとして登録する。毎回プロンプトを書く手間がなくなり、チーム全員が同じ品質の指示を出せる。

3

Plan modeで計画を立てる（第3章）
大きな変更の前に、Plan modeで設計方針と実装計画を練り上げる。AIと対話しながら計画を磨くことで、実装段階でのやり直しを減らせる。

4

Claude Code + Subagentで実装する（第4‐5章）
対話的な実装に加え、Subagentによる並列タスク実行で大規模な変更も効率的に進められる。git worktreeを組み合わせれば、複数ブランチでの同時作業も可能。

5

hooksで品質ゲートをかける（本章）
コミット前の自動テスト、ファイル書き込み後の自動フォーマットで、AI生成コードの品質を自動的に担保する。レビュー前に機械的な問題を潰せる。

6

MCP連携で外部ツールを統合する（本章）
jira、GitHub、データベースなどの外部ツールとClaude Codeを接続し、情報の参照と反映を一元化する。ブラウザとターミナルの切り替えが減り、作業の流れが途切れない。

7

ドキュメントも自動生成する（本章）
コードの変更に合わせてREADMEやAPI仕様書を自動生成し、手動で微修正してコミットする。ドキュメントとコードの乖離を防ぎ、チーム全体の情報制度を高める。

🧩
この7つのピースが組み合わさったとき、AI駆動開発は「個人のAIスキル」から「チームの開発プロセス」へと昇華する。
Mechanism ／ 仕組み

Claude Codeのhooks機能

AI生成コードに自動で品質チェックをかける、hooksの設定。どこに書き、いつ走り、どう構成するのかを順に見ていく。

設定場所
設定ファイルスコープ
.claude/settings.jsonプロジェクト単位（Git管理可能、チームで共有）
~/.claude/settings.jsonユーザー単位（全プロジェクト共通）
主なフックポイント
フック名発火タイミング用途の例
PreToolUseツール実行前lint、テスト、静的解析（matcher: Bash）
PostToolUseツール実行後自動フォーマット、import整理（matcher: Write|Edit）
設定の構造
{ "hooks": { "フック名": [ { "matcher": "対象ツール名のパターン" "hooks": [ { "type": "command", "command": "実行するスクリプトのパス" } ] } ] } }
🧩 各項目の意味
  • matcher : 対象ツールを指定する。正規表現が使える。主なツール名: Bash（シェルコマンド実行）、Write（ファイル作成）、Edit（ファイル編集）、Read（ファイル読み取り）、Glob（ファイル検索）、Grep（内容検索）など
  • hooks : 実行するフック処理の配列。type には command を指定する
  • command : 実行するシェルスクリプトのパス。スクリプトはstdin経由でJSON形式の入力データ（ツール名、入力パラーメータなど）を受け取る
  • PreToolUse でスクリプトが終了コード2を返すと、ツールの実行がブロックされる
活用のポイント

hooksを活用することで、AI生成コードに対して自動的に品質チェックをかけられる。特にチームでClaude Codeを使う場合、.claude/settings.json をGit管理して共有すれば、全員のClaude Codeに同じ品質ゲートが適用される。

Lessons ／ 学び

この章で田中さんが得たもの

1

AIの一次フィルタで機械的な問題を潰し、人間は設計判断とビジネスロジックに集中する

2

hooksはCI/CDのローカル版みたいなもので、AIの出力に品質ゲートを設けることで、レビュー前に問題を潰せる

3

Claude Codeは単体ツールではなく、MCPを通じて開発エコシステム全体のハブになる

4

ドキュメントもAIに下書きを任せて、常に最新に保つ

5

AI駆動開発は、個々の機能を知っているだけでは不十分。CLAUDE.md、Skills、Plan mode、Subagent、hooks、MCP、ドキュメント生成、これらを「ワークフロー」として設計してはじめて、チームの開発プロセスに組み込める。
