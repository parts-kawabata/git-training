# 書籍：ストーリーで覚えるClaude Code開発ガイド

## 主な登場人物
- 田中 優太（主人公）
- 佐藤 美咲（メンター）
- 河野 拓実（プロジェクトマネージャー）
- 高橋 部長

真面目で堅実な若手エンジニアである田中さんが、頼れる先輩・佐藤さんの助けを借りながら、技術的な課題を一つひとつ乗り越えていく物語

## 第8章
**あらすじ**

高橋部長との評価面談で、田中さんの取り組みは正式に認められることになります。しかし、田中さんの目に、ある記事が飛び込んできました。「AIマルチエージェントで開発チーム丸ごと自動化」AIがチーム全体の仕事を代替する未来が来たら、自分たちエンジニアの価値は何になるのか。佐藤さんとの最後の対話を通して、田中さんは何を得られるのか。そして、佐藤さんからの教えを、田中さんはどう繋いでいくのか。田中さんの物語が、いよいよ最終章を迎えます。

**田中さんが得たもの**

①ニュースが突きつける不安

田中さんが自宅でスマートフォンをスクロールしながらニュースを流し読みしていると、ある記事が目に入りました。

**「AIマルチエージェントが開発チーム丸ごとの自動化を実現ーー設計・実装・テスト・デプロイをAI群が自律分担」**

複数のAIエージェントが役割分担し、人間の介入なしにプロジェクト全体を管理する。設計エージェントがアーキテクチャを決め、実装エージェントがコードを書き、テストエージェントが品質を検証し、デプロイエージェントが本番環境にリリースする。

次の記事。

**「長時間実行タスクでAIが自律的にプロジェクト全体を管理ーー"バックグラウンドエージェント"が夜間にコードベース全体をリファクタリング」**

（AIの進化は、もっと先を行っている。自分が今やっている"AIとの協働"すら、そのうちAIだけで完結するようになるのか？自分が半年かけて身につけたことは、1年後には価値がなくなるのか？）

（自分の価値って何だ？）

②佐藤さんとの「最後の対話」

「AIがもっと進化したら...エンジニアの価値はどうなるんでしょうか。」

田中さんの問いに、いつもなら「例えばね」とすぐに話し始める佐藤さんが、珍しく間をとって答えました。

「何かきっかけがあった？」

田中さんはニュースのことを話しました。マルチエージェント、バックグラウンドエージェント、AIだけで完結するプロジェクト管理。そして、自分が半年かけて身につけたことが無価値になるのではないかという恐怖。

佐藤さんは静かに聞き、「不安を感じるのは、ちゃんと考えている証拠だよ。」と口を開きました。

「正直に言うと、"コードを書く"ということの価値は、間違いなく下がっていくと思う。そのうち、"コードを読む"ことの価値も小さくなるかもしれない。AIが読んで説明してくれるから。」

「じゃあ...エンジニアには何が残るんですか。」田中さんの問いに対する佐藤さんの考えはこうでした。

「**頭の中で、システム全体を立体的に組み立てられる力**だと思う。例えば、このサービスをここに置くとデータの流れはこうなる、3ヶ月後にユーザーが増えたらここがボトルネックになる、チームが2つに分かれたらこの境界で切るのが自然。そういう構造を、**頭の中で回せる力**。これはコードの読み書きとは別の能力。AIはコードを書ける。読める。説明もできる。でも、"このアーキテクチャはこの組織にとって正しいのか"、"半年後の変化に耐えられるか"、そういう**文脈の塊みたいな判断**は、まだ人間にしかできない。そしてたぶん、最後まで残る。」

そして佐藤さんは、"自分の手で全部書かないとダメだ"と思い込んでいたから、3ヶ月で体を壊しかけた経験があると教えてくれました。しかし、Claude Codeに出会って、**書くことを手放しても、構造を見る力があれば、エンジニアとしての自分は揺るがない**と気づいたと。

「CLAUDE.mdを書くとき、プロジェクト全体の構造を頭の中で整理した。Plan modeで設計を練るとき、コンポーネントの関係を立体的に考えた。これが、その力の入口。今すぐ完璧にできる必要はないけど、**"自分はこの方向に伸びていけばいい"と知っていること**は、それだけで大きなアドバンテージだと思う。」

田中さんの中で、何かが静かに腑に落ちました。

（コードを書く価値は下がる。読む価値も下がるかもしれない。でも、**システムを頭の中で立体的に組み立てる力**、佐藤さんが言う"構造を見る力"は、むしろこれから必要とされる。CLAUDE.mdを書いたとき、Plan modeで設計したとき、自分はもうその力を使い始めていたんだ。今はまだ入口かもしれない。でも、**どこに向かえばいいかはわかった**。）

③ツールの使い分けを整理する

| ツール | 得意な場面 |
|--------|---------------|
| GitHub Copilot | 日常的なコード補完、定型的な実装パターンの高速入力 |
| Cursor | UIベースでの対話的な探索、エディタ内でのリファクタリング |
| Claude Code | 深い対話による設計の壁打ち、大規模なリファクタリング、自律的なタスク実行 |
| ChatGPT/Claude（Web） | 一般的な技術質問、調査、概念の理解 |

すべてに共通する原則：どのツールを使っても、「何を作るか」を決めるのは自分。「品質を判断する」のも自分。

（ツールは手段。目的は良いソフトウェアを作ること。AIが進化してツールが変わっても、この原則は変わらない。）

ツールは世代交代します。5年前にはGitHub Copilotもなかった。3年前にはClaude Codeもなかった。5年後には、今は想像もつかないツールが当たり前になっているかもしれません。

でも、「問いを立てる力」と「品質を判断する力」は、ツールが変わっても持ち越せる資産です。

    AIが進化しても変わらないもの：
      1. 何を作るかを決める力
      2. それが本当に必要かを判断する力
      3. 品質に責任を持つ覚悟

④エピローグ

昼休み、田中さんのもとに、先月入社したばかりの藤井さんが声をかけてきました。

内容は、Claude Codeを使ってみたいけど、何から始めればいいか分からないというものでした。

田中さんは、設計に行き詰まり、Copilotの補完に物足りなさを感じ、「もっとプロジェクト全体を見てくれるAIツールはないのか」と思っていたあの日の自分。エレベーターホールで佐藤さんに声をかけられるまで、一人で悩んでいた自分を思い出しました。

「最初は何を聞いていいか分からないと思うけど、それで大丈夫。自分のプロジェクトのコードを見せて、『ここを改善したい』って言うだけでいいから。AIが提案してくれるから、それを自分の目でレビューして。"使える"と思ったら採用、"違う"と思ったら修正を指示する。その繰り返し。」

「"使いこなす"とはちょっと違うかもしれない。AIは道具だけど、一方的に使うんじゃなくて、一緒に考えるんだ。自分が問いを立てて、AIが手を動かして、自分が判断する。その繰り返しで、気づいたらいいコードができてる。それが"協働"っていう感じ。」

（佐藤さんがしてくれたことを、今度は自分がする番だ）

佐藤さんは答えを教えてくれたわけではありませんでした。「まず試してみて」「例えばね」「大事なのは～じゃなくて～」。いつも、田中さん自身が考え、判断し、行動するための「きっかけ」を与えてくれただけです。

そして今、田中さんが藤井さんにしたことも、同じでした。

（AIとの付き合い方は、人から人へ伝わっていく。ツールの使い方じゃなくて、"姿勢"だから。「問いを立てる」「判断する」「一歩ずつ試す」ーーこの姿勢は、ツールが変わっても、AIが進化しても、変わらない。）

    振り返りーー田中さんの半年間

    1. 第1章：はじめてのAI駆動開発ーーAIは補完ツールではなく、設計から一緒に考えてくれる「もう一人のエンジニア」。ただし最終判断は自分。
    2. 第2章：CLAUDE.mdとSkillsーーCLAUDE.mdでチームのルールをAIに教え、Skillsで繰り返す作業を定型化する。AIへのオンボーディングが品質を変える。
    3. 第3章：Plan modeとテストーーいきなりコードを書かせるのではなく、Plan modeで計画を立てる。テストの量産はAIに任せ、「何をテストすべきか」の判断に集中する
    4. 第4章：既存コードの探索とデバッグーー10万行のコードベースでも、AIに探索とデバッグを任せて自分は「判断」に集中する。Bash操作もAIの得意分野。
    5. 第5章：Subagentと並列作業ーーSubagentとgit worktreeで並列にリファクタリング。ただしレビューは怠らない。任せすぎた失敗から「信頼と検証」の線引きを学ぶ。
    6. 第6章：hooksとMCPでワークフロー設計ーーコードレビューの自動化、外部ツール連携。AI駆動開発を「個人の技」から「チームの仕組み」に昇華する。
    7. 第7章：チームへの展開ーーAI駆動開発の導入は押しつけず、成果で示す。全員が同じペースで変わる必要はない。「仕組み」を作って自走を促す。
    8. 第8章：AIと歩むエンジニアの未来ーーAIが進化しても「問いを立てる力」と「品質を判断する力」は人間のもの。ツールは変わっても、原則は変わらない。

**コラム：「手を動かす力」×「AIを使う力」= エンジニアの価値**

**AIを使わない場合：手を動かす力だけ**

自分の手でコードを書く力は重要だが、1人の人間が1日に書けるコードの量には限界がある。大規模なリファクタリングも、網羅的なテスト生成も、一人で全部やろうとすれば膨大な時間がかかる。かつての佐藤さんのように、燃え尽きてしまうリスクもある。

**AIに丸投げする場合：品質を判断できない**

AIにすべてを任せれば、大量のコードが短期間で生成される。しかし、「そのコードが本当に正しいか」「このアーキテクチャが妥当か」「業務の暗黙の前提が満たされているか」を判断できなければ、品質の保証はできない。動くけど保守できないコード、テストは通るけど本質的なケースを見落としているテスト、AIの出力を無批判に受け入れることは、むしろ危険。

**AIと協働する場合：掛け算で価値が高まる**

自分の手を動かす力（設計力、判断力、ドメイン知識）と、AIを使う力（適切な指示、レビュー、品質判断）を掛け合わせたとき、エンジニアの価値は掛け算で高まる。

    エンジニアの価値 = 手を動かす力 × AIを使う力

田中さんがこの半年で実践したのは、まさにこの掛け算。自分でSpring Bootの設計判断ができるからこそ、Claude Codeの提案をレビューできた。業務の暗黙知を理解しているからこそ、AI生成テストの「残り2割」を的確に修正できた。チームのコンテキストを持っているからこそ、CLAUDE.mdに書くべきルールを判断できた。

この「掛け算」ができるエンジニアこそ、AI時代に最も必要とされる人材。

**2026年のAIコーディングツール使い分けガイド**

**日常の補完：GitHub Copilot**

IDE内で動作するコード補完ツール。Tabキーで候補を受け入れるだけのシンプルなUXが魅力。定型的なボイラープレートコード、テストのセットアップ、コメントからの実装生成など、「次の数行」を高速に書きたい場面で力を発揮する。

**UIベースの対話：Cursor**

VS Codeベースのエディタで、AIとの対話機能が統合されている。エディタ内でコードを選択して「このコードをリファクタリングして」と指示できる直感的なUIが特徴。複数ファイルにまたがるリファクタリングも、視覚的に確認しながら進められる。

**CLIベースの深い対話：Claude Code**

ターミナルから起動するAIコーディングエージェント。プロジェクト全体のコンテキストを保持したまま、設計相談から実装、テスト、デバッグまで一貫した対話ができる。CLAUDE.mdによるルール管理、Plan modeによる計画立案、Subagentによる並列作業、hooksによるワークフロー自動化など、プロジェクト運営の基盤として機能する。

**チームの自動化：hooks + カスタムコマンド + MCP連携**

Claude Codeのhooks機能でコミット前チェックやPRレビューを自動化し、カスタムスラッシュコマンド（Skills）で定型作業を標準化。MCP連携で外部ツール（Jira、Slack、データベースなど）との統合も実現できる。

**AI駆動開発の共通原則**

どのツールを使う場合でも、以下の原則は共通。
- **「何を作るか」は人間が決める**ーーAIは実装を手伝えるが、プロダクトの方向性を決められるのは、ドメインとユーザーを理解している人間だけ。
- **AIの出力は必ずレビューする**ーーAIが生成したコードをそのままコミットしない。自分の目で読み、品質を判断する。
- **ルールはCLAUDE.mdで言語化する**ーーチームの暗黙知を明文化し、AIと人間の双方が参照できるドキュメントにする。

**結び**

半年前、田中さんは1人でAPI設計に悩んでいました。しかし今、田中さんは「もう一人のエンジニア」と一緒に設計を考え、コードを書き、品質を判断し、チームに仕組みを広げ、次の世代にその姿勢を伝えようとしています。AIは進化し続け、ツールは世代交代し、今日の最先端は、明日には当たり前になります。でも、「何を作るべきかを考える力」「それが本当に正しいかを判断する力」「一歩ずつ試しながら前に進む勇気」ーーこれらは、どんなにAIが進化しても、人間の中にしかないものです。まずは、対話をすることから始めてみよう。
