## 主な登場人物
- 田中 優太（主人公）
- 佐藤 美咲（メンター）
- 河野 拓実（プロジェクトマネージャー）
- 高橋 部長

真面目で堅実な若手エンジニアである田中さんが、頼れる先輩・佐藤さんの助けを借りながら、技術的な課題を一つひとつ乗り越えていく物語

## 第4章
**あらすじ**

Claude Codeの力を借りて、2週間の予定が1週間で動くところまで持っていくことができた田中さんに、再び新しい依頼が舞い込みました。それは、社内でも最大規模の基幹システムのリファクタリングプロジェクトに入ってほしいという内容でした。しかし、前任者は退職し、ファイルは800以上、そして、10万行にわたるコードがありました。田中さんはコードを把握しようと試みますが、あまりにも長いコードのため、うまく進めていくことができず、ただ時間が過ぎていくばかりです。自分の力だけでは2時間経っても理解することができなかった田中さんでしたが、その後、わずか15分でコードを把握することに成功します。一体どんな方法で、10万行にわたるコードを把握することができたのかー。

**田中さんが Claude Code を使って行っていたこと**

①リポジトリ全体を理解してもらう

    このリポジトリの全体構造を説明して。
    　主要なパッケージ、エントリーポイント、データフローを整理して。
    　特に重要なドメインモデルとその関連を図示して。

Claude Codeがファイル構造の探索を始め、ターミナルの画面に、ファイルを読み込んでいく様子が流れていき、数分後、回答を返した。

まず、パッケージの全体構造が図示された。core パッケージが共通基盤、order が受注管理、inventory が在庫管理、integration が外部連携。それぞれの依存関係が矢印で示されている。

次に、主要なAPIエンドポイントの一覧。どのコントローラがどのURLを受け持ち、どのサービス層を呼び出しているか。田中さんが2時間かけても掴めなかった全体像が、整理された形で表示されている。

さらに、ドメインモデルの関連図。Order → OrderItem → Product → InventoryUnit という主要なエンティティの関連と、それぞれのテーブル構成。

②実際のコードと見比べ

表示された構造図とIntelliJで実際のコードを見比べ、一つずつ確認。

パッケージ構成は正確、エンドポイントの一覧も実際のコントローラと一致している。

ドメインモデルの関連に一箇所だけ補足が必要だったが、おおむね正確。

③CLAUDE.mdの作成

    この情報をもとに、既存プロジェクト用のCLAUDE.mdを作成して。
    　パッケージ構成、命名規則、主要なアーキテクチャパターンを整理してほしい。

Claude Codeは既存コードから読み取った規約（命名規則、レイヤードアーキテクチャのパターン、例外処理の方針、テストの書き方）をCLAUDE.mdとして出力。前任者がドキュメントに残さなかった暗黙のルールが、コードから逆引きされて明文化された。

田中さんは生成されたCLAUDE.mdを一行ずつレビューし、河野PMから聞いた補足情報を書き加えた。

●3日はかかると思っていた全体構造の把握が、Claude Codeではものの15分で終わった

④スタックトレースの原因の特定

    #incident-alert 本番環境で受注一覧APIのレスポンスが断続的にタイムアウトしています。影響範囲を調査中。

スロークエリログを確認したところ、クエリの本数が異常に多く、1リクエストあたり数百本のSQLが発行されているように見える。どのコードがこれだけのクエリを発行しているのか、ログだけでは特定できず、「このあたりが怪しい」という当たりは付けたが、確信は持てないまま、2時間が経過していた。

田中さんは、APMツールからコピーしたスタックトレースとスロークエリログをそのまま貼り付けた。

    以下のスタックトレースの原因を特定して。

      [スタックトレースを貼り付け]

      本番環境で /api/orders のレスポンスがタイムアウトしている。
      スロークエリログを見ると、1リクエストあたり数百本のSQLが発行されている。関連するコードを探索し、原因と修正案を提示して。
      修正後、関連するテストを実行して影響範囲を確認して。

Claude Codeが動き始め、まずはスタックトレースに含まれるクラス名をもとに、関連するコードファイルを次々と読み込んでいく。OrderService → OrderRepository → OrderItemRepository。そして OrderService.getOrderList() メソッドの実装を詳しく分析し始めた。

1分ほどで、Claude Codeが原因を指摘した。

OrderService.getOrderList() の中で、注文一覧を取得した後、各注文に対してループで OrderItemRepository.findByOrderId() を呼び出しており、注文が500件あれば、500回のSQLが追加で発行される。典型的なN+1クエリ問題。

Claude Codeはさらに修正案も提示した。@EntityGraph アノテーションを使って関連エンティティを一括取得する方法、あるいはJPQLの JOIN FETCH を使う方法の2案。

⑤テストの実行

    関連テストを実行して影響範囲を確認して。

Claude Codeがターミナルでテストスイートを実行し始め、結果が整理されて表示される。既存のテスト32件がすべてパス。

Claude Codeはさらにテスト結果を分析し、「修正箇所に関連するテストはすべて成功しています。ただし、パフォーマンスに関するテストが存在しないため、N+1の再発を防ぐテストの追加を推奨します」とコメントを添えた。

●Claude Codeはバグの特定だけでなく、テストの実行、ログの確認、普段なら自分でターミナルを叩いて行っていた作業をそのままやってくれた。Bashコマンドを自分で打つ代わりに、「テストを実行して」と言うだけで、Claude Codeが行ってくれる。

⑥Claude Codeの提案をそのまま適用する前の確認

1つ目：JOIN FETCH でデータ量が増えた場合のメモリへの影響

　　　　→受注一覧は最大1,000件のページネーションが入っているため、問題ないと判断

2つ目：同じパターンが他のAPIでも使われていないか

      →Claude Codeに「同様のN+1パターンが他にもないか調査して」と依頼すると、InventoryService.getStockList() にも同じ問題が潜んでいることが分かった。これは今回のインシデントとは無関係だが、将来の障害を予防できる。修正を適用し、テストを実行。全テストがグリーン。田中さんはプルリクエストを作成し、ステージング環境にデプロイして動作確認を完了した。

**コラム：既存コードベースの「最初の一手」**

Claude Codeに「このリポジトリの全体構造を説明して」と聞くだけで、パッケージ構成、主要なクラス、データフロー、依存関係が整理される。もちろん100％性格とは限らないが、「全体の地図」を手に入れた状態で詳細に入るのと、地図なしで歩き回るのとでは、効率がまるで違う。

大事なのは、AIが出した「地図」を自分の目で確認すること。地図を信じて歩き出すのではなく、地図を片手に自分の足で確かめる姿勢が、既存コードベースでのAI活用の鍵となる。

**既存コードベースでのClaude Code活用**

**全体構造の把握**

新しいプロジェクトにアサインされたら、まず「このリポジトリの全体構造を説明して」と聞く。パッケージ構成、エントリーポイント、データフロー、主要な依存関係が整理される。数日かかる全体像の把握が、数十分に短縮される。

**バグの特定**

スタックトレースやエラーログをそのまま貼り付けるだけで、Claude Codeが関連コードを探索し、原因を特定してくれる。人間が目でコードを追うよりも、網羅的かつ高速に原因にたどり着けることが多い。

**CLAUDE.mdの重要性**

既存コードベースの規約、アーキテクチャの方針、命名規則をCLAUDE.mdに記載しておくと、Claude Codeの出力精度が格段に上がる。前任者がドキュメントを残していなくても、Claude Codeにコードから規約を読み取らせ、CLAUDE.mdとして明文化することも可能。

**新規と既存の違い**

| 観点 | 新規プロジェクト | 既存プロジェクト |
|------|-----------|---------------|
| 最初のステップ | 「こういうものを作りたい」 | 「このコードの全体構造を教えて」 |
| AIへの入力 | 要件・設計方針 | スタックトレース、ログ、既存コード |
| コンテキストの量 | 少ない（自分が決める） | 多い（既存コードが決めている） |
| CLAUDE.mdの役割 | コーディング規約の定義 | 既存規約の明文化+追加方針 |
| AIの主な役割 | 「一緒に作る」 | 「まず理解する」→「一緒に直す」 |

**学び**

AI駆動開発の原則は、新規でも既存でも同じ。ただし既存コードではコンテキストの量が違う。
- 全体構造の把握：「リポジトリ構造を説明して」から始める
- バグ特定：スタックトレースを貼り付けるだけ
- Bash操作の委任：テスト実行もログ確認もAIに任せる
- CLAUDE.md：既存コードの規約を明文化しておくと精度が上がる

デバッグの8割はAI、残り2割の判断は自分。その2割がエンジニアの価値。

**次回予告**

「コードベースを理解し、バグも直せるようになった。でも次は...数十ファイルにまたがるリファクタリング」河野PMから告げられた期間は2週間。「2週間で完了できるか？」という問いに、田中さんはまだ答えを持っていません。再び佐藤さんから助言をもらったことで、田中さんの開発スタイルが、さらに大きく変わる瞬間が訪れます。
