Prologue ／ あらすじ

計画を立ててから、走る。

“

「このまま本番運用に引き上げたい」——河野PMからの依頼は、田中にとって誇らしくもあり、同時に喉の奥に小さな棘を残した。プロトタイプは、確かに動く。けれど、テストはほとんど無い。サービス層のロジック、APIのバリデーション、エラーハンドリング——その全部にテストを書くとなると、一人でいったい何日かかるのか。指折り数えるほどに、ため息は深くなっていった。

抱えきれなくなった田中は、いつものように佐藤さんに相談する。すると返ってきたのは、聞き慣れない言葉だった。「“Plan mode”は、もう試した?」——いきなりコードを書かせるのではなく、まず全体の計画を立てさせるモードだという。

いつもなら、田中はすぐに作業を頼んでいた。けれど今回は、佐藤さんの助言に従って、まず計画を立てるところから始めてみた。何から手をつければいいのか分からなかったはずなのに——気づけば、最終確認とデプロイまで、わずか一週間ほどで終わっていた。

なぜ、これほど短い時間で終わらせることができたのか。そして、“Plan mode”とは、いったいどういうものなのか——。

What happened ／ 舞台裏

田中さんがClaude Codeを使って行っていたこと

本番品質への引き上げ。田中さんはいきなり手を動かさず、まず Claude Code と“計画”から始めた。その一週間を、順を追って覗いてみよう。
Claude Code とは ── 指示を日本語で書くだけで、コードを設計し、ファイルを作り、修正までしてくれる「AIの開発パートナー」のこと。

計画を立てる

【Claude Codeへの指示例】Plan modeでの改善計画立案

田中さん → Claude Code
---
 (Shift+Tab を押してPlan modeに切り替えてから入力)

このプロトタイプを本番品質にするための改善計画を立てて。
設計の問題点、テストの優先順位、リファクタリングの順序を整理して。
---

Claude Codeはプロフェクト全体のコードを読み始め、分析結果が返ってきて、問題点が3つのカテゴリに整理されていた。

🧩 問題点（3つのカテゴリ）
  • 1. APIのバリデーション不足
  • TimeEntryControllerのリクエストパラメータに型チェックしかない
  • 日付の範囲チェック、工数の上限チェックが未実装
  • 不正な入力でサーバーエラー（500）が返る箇所が複数
  • 2. エラーハンドリングの統一
  • 各メソッドでtry-catchの書き方がバラバラ
  • カスタム例外クラスが定義されているが使われていない箇所がある
  • 3. サービス層のテスト不足
  • TimeEntryServiceのビジネスロジックにテストがゼロ
  • 月次集計ロジック（calculateMonthlyTotal）が最も複雑だがテストなし

さらに改善の優先順位まで提示してくれた。

【改善の優先順位（Claude Codeの提案）】 1. TimeEntryService のビジネスロジックテスト（最重要） → ここが壊れると全機能に影響。最優先でテストを整備 2. バリデーション追加 + バリデーションテスト → 不正入力で500エラーが出る状態は本番では許容できない 3. エラーハンドリングの統一 → GlobalExceptionHandler に集約するリファクタリング 4. Controller層の統合テスト → API全体の動作確認

自分1人で考えていたら、「とりあえずControllerからテストを書くか」と表面的なところから手をつけていたかもしれないところを、Claude Codeは、コード全体を分析した上で「ビジネスロジックのテストが最優先」と指摘してくれた。

🔰 はじめての人へ ─ ここに出てくる言葉
  • Plan mode：すぐにコードを書かせず、まず分析と計画だけをさせる Claude Code のモード。Shift+Tab で切り替える。
  • バリデーション：入力された値が正しいかどうかを検査する仕組み。
  • サービス層／ビジネスロジック：アプリの“本体の判断”を担う中心部分。ここが壊れると全機能に影響する。
  • リファクタリング：動きは変えずに、コードの形をきれいに整え直すこと。

計画に沿って、核となるビジネスロジック（TimeEntryService.calculateMonthlyTotal）のテスト

指定した月のユーザーの工数を集計するメソッドで、プロトタイプの中で最も複雑なロジック

【Claude Codeへの指示例】テスト生成の具体的な指示

田中さん → Claude Code
---
TimeEntryService.calculateMonthlyTotal のテストを書いて。
正常系3パターン、異常系2パターン、境界値1パターン。
JUnit5 + Mockitoで。CLAUDE.mdのテスト規約に従って。
---

Claude Codeは6つのテストメソッドを生成した。

正常系の3パターン：「通常の月次集計」「工数ゼロのユーザー」「複数プロジェクトにまたがる工数の合算」どれも的確。テストデータのセットアップも、第2章で整備したCLAUDE.mdのテスト規約に従っている。

異常系の2つ目で、「不正な月（13月）を渡した場合」、テストはIllegalArgumentExceptionがスローされることを期待しているが、実際のコードには月のバリデーションがまだ実装されていない。

🔰 はじめての人へ
  • JUnit5：Javaでテストを書くための定番ツール。
  • Mockito：本物の代わりに“ニセモノ部品”を用意して、対象だけを切り出して試すための道具。
  • 正常系／異常系／境界値：うまくいく場合・失敗する場合・ぎりぎりの値（0や上限など）を、それぞれ確かめるテストの観点。

テストコードの修正

境界値テストでは、日付の計算に使うYearMonthクラスの扱いが実際のコードと少しずれていたため、テストコードを修正し、境界値の条件を微調整する。

6つ中5つはそのまま使え、1つは修正が必要。続けてcreateTimeEntry、updateTimeEntry、deleteTimeEntryのテストも生成させていく。

2割は修正が必要で、業務ロジック固有の前提条件（「工数の最小単位0.5時間」「同一プロジェクトの同日エントリーは合算」など）は、コードに明示されていない暗黙の仕様だったため、Claude Codeが正確に把握できなかった。

🔰 ポイント
  • AIが生成したテストもそのまま信じない。実際のコードと突き合わせ、ずれた前提を人間が直す──ここに自分の価値がある。
⚡
Claude Codeが生成するテストの約8割はそのまま使えるもの。基本的なCRUD操作のテスト、バリデーションの検証、明確な例外処理のチェックなど、定型的なパターンはAIの得意分野。
Column ／ コラム

AI生成テストの「8割/2割」法則

AIが書いたテストは、どこまで信じていいのか? その線引きを「8割」と「2割」で分けて考えると、人間の出番がくっきり見えてくる。

✅ 8割：そのまま使えるテスト

・基本的なCRUD操作の正常系・異常系
・バリデーションのチェック（null、空文字、型不一致）
・明示的な境界値テスト（0、最大値、リストの空）
・フレームワーク固有のパターン（Mockitoのセットアップ、JUnit5のアサーション）

🛠️ 2割:修正が必要なテスト

・業務固有のロジック（「工数の最小単位は0.5時間」など、コードに明示されていないルール）
・暗黙の前提条件（「このメソッドは必ずトランザクション内で呼ばれる」など）
・複数の条件が組み合わさるエッジケース

「8割をAIに任せて、2割の判断に集中する」ことが、AI協働テストの基本戦略。AIにテストの「量」を任せることで、人間は「質」の判断に集中できる。

Plan Mode ／ 活用法

Plan modeの活用法

いきなり書かせない。まず計画を立てる──このひと手間が、一週間という結果を生んだ。Plan modeの正体と、使いどころを整理しておこう。

Plan modeとは

Claude Codeに「まず計画を立てて、まだコードは変更しないで」と伝えるモード。通常のClaude Codeは指示を受けるとすぐにコード変更を提案・実行するが、Plan modeでは「分析」と「計画立案」だけを行う。

使い方

Shift+tab を押してPlan modeに切り替えてから指示を出す。通常モードに戻すときも、もう一度 Shift+Tab を押す。

(Shift+Tab でPlan modeに切り替え) このプロジェクトのアーキテクチャ上の問題点を分析して。 (Shift+Tab でPlan modeに切り替え) テスト戦略を立てて。優先順位をつけて。 (Shift+Tab でPlan modeに切り替え) このモジュールのリファクタリング計画を作って。
Plan modeが有効な場面
設計の見直し
プロトタイプから本番品質への引き上げ
リファクタリング計画
大規模な構造変更の前に影響範囲を分析
テスト戦略の策定
どのモジュールから、どの優先順位でテストを書くか
技術的負債の棚卸し
コードベース全体の問題点を洗い出す
Plan mode→承認→実行のワークフロー
ステップ1: Plan modeで計画を立てる Shift+Tab でPlan modeに切り替え → 指示を入力 → Claude Codeが計画を提案 ステップ2: 計画をレビューし、承認する 人間が計画を確認 → 必要に応じて修正を依頼 ステップ3: 承認した計画に沿って実行する 計画の各項目を順番に実行 → 1つ完了するごとに確認

AIが「何をするか」を人間がコントロールし、闇雲にコードを変更させるのではなく、全体の方向性を合意してから実行する。これがPlan modeの設計思想。

Lessons ／ 学び

この章で田中さんが得たもの

1

テストが先にあることで、「実装すべきバリデーション」が明確になる。テストを書いてから実装を追加するのは、テスト駆動開発の考え方。

2

テストの量産はAIが得意だが、「何をテストすべきか」を判断するのは自分。AIが見落とす暗黙の仕様、業務ロジックの前提条件を修正することに、自分の価値がある。

3

「計画→テスト→実装。計画→テスト→実装。」これはTDDの変形。テストが通るように実装する開発手法。

4

「何をテストすべきか」の判断は自分がして、「テストコードの実装」はAIが拘束に行う。そしてテストが通るように実装を改善する。この分業が、自然とTDD的なワークフローを生み出していた

【開発ノート】AI協働のPDCAサイクル
Plan: Plan modeで全体計画を立てる（AIと一緒に） Do: テストをAIに生成させる → 実装を改善する Check: テストを実行し、品質を確認する Action: 計画を修正し、次のサイクルへ
