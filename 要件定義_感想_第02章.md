# 第02章「要件定義の基本的な流れ」感想文
1. この章の要点（自分の言葉で3行）：要件定義の基本的な流れは、まず、心の中で望んでいる状態から目に見える言葉にすることで、要望から要求に変わる。そして、その要求を依頼された側が検討し、修正が必要な案には代替案を考え、依頼主に提案する。その繰り返しでお互いに合意したときに要件になる。
2. なるほど／腑に落ちた点（1〜2個）：代替案は、「このままではできない」というときだけに考えるのではなく、そのままで十分可能な場合だとしても、それよりもより良い案があれば、代替案として提示した方がよい。たいていの依頼主は、専門家の知見が加味されることを期待するものだから、いつでも代替案を考えておくとよい。
3. 身近な例で言いかえると（教本の例、または「入金消込」のような“想像でわかる”例でOK。実際の自社業務は知らなくてよい）：掃除をお願いするときに、「ここを綺麗にしてほしい」「この場所はこのやり方で掃除してほしい」という要望を紙にまとめて、掃除をする人に紙を渡して「この順番・やり方で掃除して」と要求する。もし、掃除する側が、記した通りに行うのが難しい場合は、「ここのやり方を変更して」と修正をお願いする。その修正をもとに掃除をお願いする側はもう一度考え直し、新しいやり方を提案する。それを繰り返して、お互い合意した掃除の仕方ができたら、それが条件となり、掃除する人はその条件通りに掃除をしていく。
4. 疑問・もっと知りたい点（1個）：代替案を考えるとき、どんなことを基準にして考えるのか
5. 前の章・V字（開発工程）とのつながり（1行）：要件定義のなかの要件をまとめるまでの流れ

①「要望」が「要件」になるまでに何が起きる？
- 「こうしたい」という要望を、言葉にすることで要求になり、その要求を依頼主も依頼された側も納得する内容になるまで検討・提案し、お互いが合意してできたものが要件になる。

②なぜ“合意”が要るのか（勝手に決めるとどうなる）？
- 勝手に決めてしまうと後に、依頼主は「依頼していた内容と違う」となり、依頼された側は「このやり方ではできない」となってしまい、お互いが納得できないものになって一からやり直しとなる可能性が高いため、最初に合意する要件を作っておかなければならない。
