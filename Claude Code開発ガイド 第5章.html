Prologue ／ あらすじ

二週間と、数十ファイルの宿題。

“

コードベースを理解し、バグも直せるようになった。それでも田中の前には、まだ重い宿題が積まれていた。数十ファイルにまたがるリファクタリング。テストはある。けれど、一人で全部を終わらせるには重すぎる。「——二週間で、間に合うのか」。確信は、どこにもなかった。

再び、田中は佐藤さんのもとへ足を運ぶ。事情を打ち明けると、佐藤さんは二つの言葉を差し出した。「“Subagent”と、“git worktree”」。それを使えば、複数の作業を並列に進めていける——そう、佐藤さんは言う。

教わったとおり、田中は作業を走らせ始めた。複数のタスクが同時に動き、影響範囲の分析が並んで進んでいく。順調だった。あまりにも、順調だった。——そのとき、ひとつの過ちが、静かに混ざり込んでいたことに、田中はまだ気づいていない。

果たして、その失敗をうまく乗り切ることができるのか。そして、二週間という期限に、間に合うのか——。

What happened ／ 舞台裏

田中さんが Claude Code を使って行っていたこと

並列で動かし、慌てず捌き、そして自分の手で確かめる。田中さんと Claude Code のやり取りを、順を追って覗いてみよう。
Claude Code とは ── 指示を日本語で書くだけで、コードを設計し、ファイルを作り、修正までしてくれる「AIの開発パートナー」のこと。

Subagentを意識した指示

まずはgit worktreeの準備。リファクタリング専用の作業ディレクトリを作る。

# リファクタリング用のworktreeを作成 git worktree add ../project-refactor feature/refactor-order # 作成されたworktreeを確認 git worktree list

ターミナルに2つのディレクトリが表示され、メインのディレクトリは main ブランチのまま。何かあればすぐにhotfixブランチを切れる状態であることを確認。

田中さん → Claude Code
OrderServiceモジュールのリファクタリングを段階的に実行して。
 1. まず影響範囲を分析して
 2. 次にインターフェースを統一して
 3. 最後に実装を整理して
各段階の結果を報告して。

Claude Codeが動き始め、ターミナルの表示が変わり、Task toolを使って内部で複数のSubagentが立ち上がっている様子が見える。

普段のClaude Codeは1つの処理を順番に進めていくが、今回は複数のタスクが同時に走っている。ターミナルには各Subagentの進捗がリアルタイムで表示され、影響範囲の分析と依存関係の洗い出しが並列で進んでいき、数分後、最初の報告が返ってきた。

🔰 はじめての人へ ─ ここに出てくる言葉
  • リファクタリング：動きは変えずに、コードの形をきれいに整え直すこと。
  • git worktree：1つのリポジトリを、複数のフォルダに同時に開いておける仕組み。フォルダごとに別のブランチを触れる。
  • ブランチ：作業を枝分かれさせておく場所。main は本番用、hotfix は緊急修正用、といった具合に分ける。
  • Subagent（Task tool）：Claude Code が内部でもう一人の自分を起ち上げ、複数の作業を同時に進める機能。
  • インターフェース：部品どうしが繋がるための“約束ごと”（メソッドの名前・引数・戻り値の形）。

追加の指示

田中さん → Claude Code
この変更はORDER_STATUSのenum定義にも影響するから、そっちも確認して」

Claude Codeはすぐにenum定義の確認をSubagentに追加し、関連する変更箇所も洗い出してくれた。そして、インターフェースの統一が完了し、田中さんはGit diffで変更内容をざっと確認し、コミットした。

🔰 はじめての人へ
  • enum（列挙型）：「注文中／発送済／キャンセル」のように、決まった選択肢の集合を表す型。
  • Git diff：変更前と変更後の差分を一覧で見せてくれるコマンド。コミット前のレビューに使う。
  • コミット：変更内容に区切りをつけて記録として残す操作。

至急の対応

田中さん → Claude Code
@channel 本番のログイン画面が崩れています。至急対応お願いします。

以前の田中さんなら、リファクタリングの途中でstashして、mainに切り替えて、終わったら戻してと、そのたびに「さっきどこまでやったっけ」と混乱していたが、今は違う。

田中さんはリファクタリングの作業ディレクトリをそのまま残し、メインのディレクトリに移動。ログイン画面のCSSが最近のデプロイで崩れていた問題を特定し、修正。テストを実行して問題ないことを確認し、PRを作成して完了。わずか約30分で終了した。

hotfixが片付いたら、リファクタリングの作業ディレクトリに戻るだけで、作業状態はそのまま、何も失われていない。

🔰 はじめての人へ
  • stash：作業途中の変更を一時的に棚上げしておく機能。ブランチを切り替えるときに使うが、戻し忘れて混乱しがち。
  • hotfix：本番で起きた不具合を、緊急で直すための作業（やそのブランチ）。
  • PR（プルリクエスト）：「この変更を取り込んでほしい」とレビューを依頼する仕組み。

想定外の変更を発見

成果をレビューするためにターミナルを開くと、OrderRepositoryの findByCustomerId メソッドのシグネチャが変わっていて、戻り値がList<Order>から Optional<List<Order>> に変更されていた。

インターフェース統一の一環として、Claude Codeが「統一的に」Optionalで包んでしまっていた。OrderService内部では問題なく動いており、テストも通っているが、リファクタリングの対象外だったPaymentServiceのテストには、OrderRepositoryの findByCustomerId を直接呼び出すケースが含まれていない。

案の定、PaymentService.java の中で findByCustomerId の戻り値をList<Order>として受け取っている箇所があり、コンパイルは通っているが、実行時に型の不整合が起きる可能性がある。

田中さんはClaude Codeに修正を指示。PaymentService の呼び出し箇所を Optional に対応させ、テストも追加。さらに、同じ問題を繰り返さないよう、CLAUDE.mdに「リファクタリング時は対象モジュールの呼び出し元への影響を必ず確認すること」というルールを追加した。

🔰 いま起きたこと（原文の解説）
  • シグネチャ：メソッドの“顔つき”＝名前・引数・戻り値の型のこと。ここが変わると呼び出し側にも影響する。
  • Optional<T>：「値があるかもしれないし、無いかもしれない」を表す入れ物。中身を取り出すには一手間かかる。
  • コンパイルは通るが実行時に不整合：型の包み方が食い違うと、ビルドは成功しても動かした瞬間に壊れる恐れがある。
  • CLAUDE.md：プロジェクトのルールを書いておくと、Claude Code がそれを踏まえて動いてくれる“約束ごと”のファイル。
🔁
この日から、田中さんはSubagentの実行後に必ず以下のルーティンを踏むようになった。
🧭 Subagent実行後のルーティン
  • 1. git diff --stat で変更ファイルの全体像を把握する
  • 2. 変更の「意図」と「実際」を照合する
  • 3. 対象外モジュールへの影響を grep で確認する
  • 4. コミットを小さく区切り、1コミット=1変更にする
Column ／ コラム

AI出力のレビューチェックリスト

並列で速く動けるようになったぶん、見落としも生まれやすい。AIの成果を世に出す前に、自分の目で通すべき5つの観点を並べてみよう。

✅ 変更範囲は指示通りか（スコープの確認）
  • 指示した範囲だけが変更されているか。意図しないファイルが含まれていないか。
✅ 意図しないシグネチャ変更はないか
  • メソッドの引数型、戻り値型、アクセス修飾子が勝手に変わっていないか。
✅ テスト対象外のモジュールへの影響は？
  • 変更したモジュールを参照している他のモジュールに影響はないか。grep で呼び出し箇所を確認する。
✅ コミット単位は適切か（1コミット=1変更）
  • 複数の変更が1つのコミットに混ざっていないか。後から切り戻しやすい粒度になっているか。
✅ CLAUDE.mdの規約に沿っているか
  • 命名規則、例外処理のルール、アーキテクチャ方針など、プロジェクト固有のルールが守られているか。

このチェックリストは完璧ではない。プロジェクトに応じて項目を追加・修正する。大切なのは「チェックリストを持つこと」そのもの。レビューの習慣化が、AI協働の品質を担保する。

Mechanism ／ 仕組み

Subagent（Task tool）とgit worktree

Subagent（Task tool）とは

Claude Codeが内部で別のエージェントを起動し、並列にタスクを実行する機能。ユーザーが「段階的にやって」「複数の観点で分析して」と指示すると、Claude Codeは自律的にTask toolを使い、複数のSubagentを立ち上げる。

🔍影響範囲分析
対象モジュールを参照している全ファイルを洗い出す
✅テスト追加
既存コードに対するテストケースを並列で生成する
✍️コード書き換え
インターフェースの統一や実装の整理を段階的に実行する

Subagentは「指示された範囲」では正確に動作するが、「指示されていない範囲」への影響は自動ではカバーしきれない。必ずレビューとセットで使うことが重要。

git worktreeとは

1つのリポジトリで、複数のブランチを別のディレクトリに同時展開するGitの機能。

# worktreeの追加（新しいブランチを別ディレクトリに展開） git worktree add <path> <branch> # 例: リファクタリング用 git worktree add ../project-refactor feature/refactor-order # worktreeの一覧表示 git worktree list # worktreeの削除（作業完了後） git worktree remove <path>

通常のブランチ切り替え（git checkout）では、1つのディレクトリで1つのブランチしか作業できないところを、worktreeを使えば、ディレクトリごとに異なるブランチを開けるため、リファクタリングとhotfixを同時並行で進められる。

組み合わせの威力

worktreeで作業ディレクトリを分離+各ディレクトリでClaude Codeを起動＝並行作業が可能

これにより「リファクタリング中にhotfixが入っても慌てない」「複数の大きな変更を独立して進められる」という開発体験が実現するため、stashで退避→復元を繰り返す日々とはお別れ。

Lessons ／ 学び

この章で田中さんが得たもの

1

Subagent（Claude CodeのTask tool）を使えば、Claude Codeが内部で別のエージェントを起動して、複数の作業を並列で進められる

2

git worktreeは同じリポジトリの別ブランチを、別のディレクトリとして同時に開ける

3

worktreeのおかげで一切慌てることなく、stashで中途半端に退避する必要もない。Subagent、worktreeはもう”AIアシスタント”ではなく”AIチームメンバー”。

4

AIに任せっぱなしにしてはいけない。AIは指示された範囲では正確だけど、”指示されていない範囲”への影響まではカバーしきれない。だから、レビューをサボってはいけない。

5

Plan modeで計画を立て、テストを生成し、Subagentで並列にリファクタリングし、レビューで品質を確認するサイクルの中でも、一番大事だったのは”レビューを怠らない”という教訓
