Prologue ／ あらすじ

二週間で、デモを。

“

「2週間で、デモを。」——高橋部長のその一言が、田中の机にそっと、しかし重く置かれた。チームの工数管理ツールのプロトタイプ。設計、実装、テスト。本来なら数人がかりの工程を、たった一人で、二週間で仕上げる。まっさらなプロジェクトを前に、田中の指はキーボードの上で止まっていた。「全部を一人で、この期間で……難しいかもしれない」。設計の入り口で、ため息が夕暮れのオフィスに溶けていく。

そのとき、穏やかな声がした。社内で“AIペアプロの佐藤さん”と呼ばれるシニアエンジニアだった。事情を打ち明けた田中に、佐藤さんは一つの名前を教える。「“Claude Code”は、もう試した?」。

使い方をさっと教わり、半信半疑のまま、田中は自分のPCでそれを動かしてみた。——その日のうちに。あれほど悩んでいた設計は形になり、基本APIの動作確認まで、すべて終わっていた。

なぜ田中さんは、Claude Codeを使っただけで、あんなに悩んでいた設計をここまで素早く終わらせることができたのか。そして、そのとき——画面の向こうでは、何が行われていたのか。

What happened ／ 舞台裏

田中さんがClaude Codeを使って行っていたこと

あの数時間、田中さんは Claude Code と“対話”しながら開発を進めていた。やり取りを、順を追って覗いてみよう。
Claude Code とは ── 指示を日本語で書くだけで、コードを設計し、ファイルを作り、修正までしてくれる「AIの開発パートナー」のこと。

最初の指示 ─ プロジェクトの骨組みを頼む

田中さん → Claude Code
Spring Bootで工数管理ツールのREST APIを作りたい。
まず、プロジェクトの雛形を作成してほしい。

要件：
- Java 17 + Spring Boot 3.x + Gradle
- エンティティ： Project, Task, TimeEntry, User
- 基本的なCRUD APIをProjectから順に実装
- データベースはH2（プロトタイプ用）

Claude Codeがディレクトリ構成、依存関係、エンティティの設計など、プロジェクト構造を提案し、確認を求めてきた。

🔰 はじめての人へ ─ ここに出てくる言葉
  • REST API：アプリ同士がデータをやり取りするための“窓口”。
  • Spring Boot：Java でこの窓口（サーバー）を手早く作るための定番の道具。
  • エンティティ：データベースの1行を表すデータの“入れ物”（例：Project, Task…）。
  • CRUD：作成・読取・更新・削除という、データ操作の4つの基本。
  • H2：お試し用に手軽に使える軽量データベース。

内容を確認後、一箇所だけ修正

田中さん → Claude Code
TimeEntryのフィールドに「description」を追加して。何の作業をしたか記録できるようにしたい。

修正が反映された設計を承認すると、Claude Codeがコードを作成し始め、エンティティクラス、リポジトリ、サービス、コントローラ、ファイルが次々とプロジェクトに追加されていく。

🔰 ポイント
  • 全部を任せきりにせず、提案を見てから一箇所だけ直す──このキャッチボールが Claude Code の使い方の肝。

レビュー中に気付く ─ バリデーションが入っていない

生成されたコードをレビューしていると、バリデーションが入っていないことに気付く。

田中さん → Claude Code
TimeEntryのhoursフィールドにバリデーションを追加してほしい。
 - 0より大きく、24以下であること
 - nullは不可

Claude Codeは既存のエンティティクラスを読み込んだ上で、@NotNull、@DecimalMin、@DecimalMaxアノテーションを追加し、コントローラにも@validを追加。関連するテストコードも一緒に提案してくれた。

🔰 いま起きたこと（原文の解説）
  • 1. 既存のエンティティクラス（データベースのテーブル1行を、Javaのオブジェクトとして表現したクラスのことで、「@Entity」を付けることで、フレームワークがこのクラスをデータベースと結びつけて管理してくれる）を読み込む
  • 2. フィールドにバリデーションアノテーション（「このフィールドはこういう値でなければならない」というルールを、フィールドの上に宣言として書く仕組み）を追加
  • 3. コントローラーに「@Valid」を追加（コントローラのメソッド引数に「@Valid」を付けて初めて、リクエスト受信時に自動でチェックが走る）

基本的なCRUD APIが動き始め、田中さんはcurlコマンドでエンドポイントを叩いて動作確認をしている。

🔰 ひとこと
  • curl：作ったAPIに手元から試しにアクセスして、ちゃんと動くか確かめるためのコマンド。

レスポンスのJSON形式をもう少し整理したい

田中さん → Claude Code
APIレスポンスを統一的な形式にリファクタリングしてほしい。
以下の形式で：
{
  "status": "success",
  "data": { ... },
  "massage": null
}
エラー時は：
{
  "status": "error",
  "data": null,
  "massage": "エラーの詳細"
}

Claude Codeは共通のレスポンスラッパークラスを作成し、既存の全コントローラを一括で修正、グローバル例外ハンドラも追加された。

🔰 はじめての人へ
  • JSON：アプリ同士がデータをやり取りするときの、世界共通の書式。
  • リファクタリング：動きは変えずに、コードの形をきれいに整え直すこと。
  • レスポンスを統一：成功でもエラーでも“同じ型”で返すと、受け取る側がとても扱いやすくなる。
⚡
田中さんは半日で、ProjectとTaskのCRUD API、TimeEntryの登録・取得・更新・削除、そして統一されたレスポンス形式まで、動作確認を完了していた。
Column ／ コラム

「補完ツール」と「エージェント」の違い

なぜ Claude Code はここまでできるのか? 似て非なる2種類のAIを並べると、その正体が見えてくる。

⌨️ 補完ツール（Copilotなど）

今開いているファイルの文脈から「次の行」を予測する。エディタの中で動作し、人間が書いているコードの延長線上を提案するのが役割。優秀な予測変換のようなもので、タイピングの手間を減らしてくれる。

🤖 エージェント（Claude Codeなど）

プロフェクト全体を理解したうえで「設計→実装→テスト」を一貫して担う。指示を受けて自律的にタスクを遂行し、複数ファイルをまたいだ変更も行える。「次の1行」ではなく「次にやるべきこと」を考えてくれる。「設計から一緒に考えてほしい」という場面では、エージェント型のツールが圧倒的に力を発揮する。

Features ／ 特徴

Claude Codeの主な特徴

📂プロジェクト全体の読み込み
ディレクトリ内のファイルを自動的に認識し、コードベース全体を把握した上で回答する
🧠対話コンテキストの保持
一連の対話の流れを記憶し、前の指示を踏まえた一貫した提案を行う
✍️コード生成 + 編集
新規ファイルの作成だけでなく、既存ファイルの修正・リファクタリングも実行する
✅テスト生成
実装コードに対応するテストコードを自動生成する
⌨️ターミナルコマンド実行
ビルド、テスト実行、Git操作などをターミナル上で直接実行できる
Lessons ／ 学び

この章で田中さんが得たもの

1

AIの提案を鵜呑みにするのではなく、自分の目で確認し、レビューする

2

修正したい箇所は具体的に、どう変更したいのかを記述する

3

一から全てAIに任せるのではなく、設計方針や条件など、大元となる内容は具体的に言語化して、自分で指定することが大事

Next ／ 次回予告

動く。けれど、“チームのコード”ではなかった。

“

自信を持って送り出したプルリクエスト。だが返ってきたのは、ずらりと並んだレビューコメントだった。動く。確かに、コードは動く。けれど——それは“チームのコード”ではなかった。作法が、ルールが、どこか少しずつ違う。修正すべき箇所は、思った以上に多かった。

かといって、毎回プロンプトで規約を一から伝えるのは現実的ではない。そのたびに同じ説明を書き、同じ時間が溶けていく。

——AIに、チームのルールそのものを覚えさせる方法は、ないのか。
