Prologue ／ あらすじ

便利だから使って、では人は動かない。

“

ワークフロー全体を、自分の手で設計できるようになった——その手応えを噛みしめていた田中の机に、新しい号令が下りてきた。「チーム全体でAI駆動開発を推進せよ」。高橋部長の言葉には、期待がこもっていた。

定例ミーティング。これまでの成果を携え、田中はAIを使った業務の進め方を提案した。手応えはある、きっと前向きに受け止めてくれる——そう信じていた。だが、会議室の空気は、思い描いたものとは違った。

ベテランの木村さんは静かに言った。「今のやり方で回っているんだよね」。入社二年目の山田さんは眉を曇らせた。「設定が難しそうで……」。そして高橋部長の問いが、机を横切る。「セキュリティは大丈夫なのか」。

自分は便利だと思っていても、全員がそう思うわけではない——田中は、提案の仕方を間違えたことに気づく。後日、佐藤さんが穏やかに告げた。「押しつけると、逆効果だよ」。

号令でも、正論でもなく。果たして田中さんは、チームのひとりひとりに、どうやって納得してもらうのか。

What happened ／ 舞台裏

田中さんがチームに働きかけて行っていたこと

「チーム全体でAI駆動開発を推進せよ」──部長の号令を受けて、田中さんが動き出す。正面からの提案、淡々とした成果の見える化、セキュリティガイドラインづくり、そしてAgent Teamsの実演まで。その舞台裏を、順を追って覗いてみよう。

定例ミーティングでの提案

「1つ目は、CLAUDE.mdの共有化です。プロジェクトルートにCLAUDE.mdを配置して、コーディング規約やアーキテクチャの方針をチーム全員のClaude Codeに共有します。これにより、誰がAIと作業しても、同じルールに基づいたコードが生成されます。」

「2つ目は、カスタムスラッシュコマンドの整備です。コードレビューやテスト生成の定型作業を、チーム共有のコマンドとして定義します。新しいメンバーが入っても、すぐに同じワークフローで作業できるようになります。」

「3つ目は、hooks設定の統一です。コミット前のチェックやPR作成時の要約生成など、品質を担保する仕組みをチーム全体に導入します。」

説明を終えた田中さんは手応えを感じ、きっとチームも前向きに受け止めてくれるはずと期待していたが、会議室の空気は違った。

最初に口を開いたのは、入社12年目のベテラン、木村さん。

「うーん、田中くんの成果は見てるよ。すごいと思う。ただ正直なところ、今のやり方で回ってるんだよね。新しいツールを覚える時間を取るよりも、今のプロジェクトの納期を優先したいんだけど。」

次に、入社2年目の後輩、山田さん。

「田中さん、興味はあるんですけど...CLIで作業するのって、設定が難しそうで。ターミナルに慣れてないと、最初のハードルが高くないですか？」

そして、高橋部長。

「セキュリティは大丈夫なんだろうな？AIにソースコードを読ませるってことだろ。うちのコードが外部に漏れるリスクはないのか。経営層にも説明できるようにしておかないと。」

💡
木村さんが「今のやり方で回っている」と言ったのは事実。納期に追われるチームに、新しいツールの学習ソフトを負担させようとしていた。山田さんの不安も正当。CLIベースのツールは、GUIに慣れた人にはハードルが高い。そして高橋部長のセキュリティの懸念は、組織のリーダーとして当然の問いかけ。自分は便利だと思っても、全員がそう思うわけではなく、便利だから使って、では人は動かない。この提案の仕方では押しつけてしまっている。

成果の見える化

チーム全員への提案はやめて、自分の日々の作業記録を淡々とSlackに投稿し始めた。チャンネル名は #ai-dev-log 。説明文には「田中の個人的な作業記録です。興味のある方はご自由にどうぞ。」とだけ書いた。

田中さんはあえて「みんなも使ってください」と書かず、佐藤さんの教え通り、自分の記録を淡々と共有するだけ。

最初の3日間、リアクションはゼロ。田中さんは焦りを感じつつも、投稿を続けた。

金曜日の夕方、最初の変化が起きた。後輩の山田さんからDMが届き、翌週、山田さんと隣同士でClaude Codeを使ってテストを書いてみた。

その翌週には、木村さんがSlackのスレッドに初めてリアクションをつけた。

3週間が経つ頃には、田中さんが「使ってください」と一度も言わないまま、#ai-dev-log のメンバーがチームの半数に増えていた。

Claude Codeでセキュリティガイドラインの作成

田中さん → Claude Code
チーム向けのAI活用セキュリティガイドラインを作成してほしい。
  以下の観点をカバーして:
  - AIに入力してはいけない情報のリスト
  - 利用可能ツールの承認プロセス
  - AI出力の確認フロー
  社内の中規模開発チーム（10名程度）を想定。

Claude Codeが生成した下書きを、田中さん、佐藤さんの2人でレビュー。

「この"入力禁止情報"のリスト、うちの社内規定に合わせて修正しよう。」

「"認証キーやAPIキー"は当然として...本番環境の接続情報も入れておいた方がいいですね。」

「あと、非公開のビジネス戦略情報。来期の事業計画とか、M&Aの話とか。コードには直接関係ないけど、AIに入力するリスクはある。」

2時間かけて、ガイドラインが完成。

高橋部長にガイドラインを提出した結果、これでチームに展開することに決まった。

🛡️
Claude Codeが作った骨子に、自分たちの業務知識を肉付けした感じ。ガイドラインを作ること自体がAIとの協働の実践例で、ルールがあれば、安心して使える人が増える。

Column ／ コラム

Subagentで十分？ Agent Teamsを使うべきとき

同じ作業の手分けか、役割の異なる協働か。たった一つの問いで、選ぶべき道具が見えてくる。

「役割分担が必要かどうか」でSubagent（Task tool）とAgent Teams、どちらを使うか決める

Subagentが向いているケース
🧩 Subagent（Task tool）
  • 同じ種類の作業を複数のファイルやモジュールに対して並列実行したいとき
  • 例：「5つのサービスクラスのテストを同時に生成して」「3つのモジュールのREADMEを並列で更新して」
  • 1つの親エージェントが全体を把握しているので、シンプルに使える
Agent Teamsが向いているケース
👥 Agent Teams
  • 異なる専門性を持つ作業を組み合わせて1つの成果物を作りたいとき
  • 例：「既存コードを調査してから、その結果に基づいて実装とテストを並列で進めて」
  • タスク間に依存関係がある（調査が終わらないと実装に入れない、など）
  • 作業の途中でエージェント間の情報共有が必要
迷ったらSubagentから始める

実務では、まずSubagentで十分かを考える。「同じことを複数回やる」ならSubagent、「違うことを組み合わせる」ならAgent Teams。田中さんも第5章のリファクタリングではSubagentで十分だったが、調査・実装・テストという異なる役割が必要な場面ではAgent Teamsが力を発揮した。

Column ／ コラム

チーム導入で避けるべき3つのアンチパターン

よかれと思った一手が、定着をいちばん遠ざける。導入でつまずきやすい3つの落とし穴。

1

強制導入

「来月からチーム全員Claude Codeを使うこと」と号令をかけるパターン。義務化された瞬間、人は心理的に反発する。特に、既存のワークフローに慣れているベテランメンバーにとっては、「今のやり方を否定された」と感じてしまう。

佐藤さんが経験したように、強制的に研修を実施しても、1か月後にはほとんどの人が使わなくなる。「やりたい人からどうぞ」のスタンスで、自発的な採用を促すほうが、定着率は圧倒的に高くなる。

2

ツール信仰

「Claude Codeを使えば全て解決」という過度な期待。AIツールは万能ではない。得意な作業（定型的なコード生成、テストの量産、パターン化しやすいリファクタリング）と、苦手な作業（業務固有の判断、暗黙の仕様の理解、組織内の政治的な調整）がある。

ツールへの過度な期待は、期待通りにいかなかったときに大きな失望に変わる。「万能ではないが、使い方次第で強力な武器になる」という現実的な期待値を、チーム内で共有しておくことが大切。

3

セキュリティ無視

ガイドラインなしにAIツールを使わせるパターン。「まずは自由に試してみよう」という姿勢自体は良いのですが、最低限のルールがないと、機密情報の漏洩やセキュリティインシデントのリスクが高まる。

一度インシデントが起きると、「やっぱりAIは危険だ」という空気が広がり、チーム全体のAI活用が後退する。田中さんが高橋部長の懸念に応えてガイドラインを策定したように、セキュリティの仕組みを先に整えることで、安心して試せる環境を作る。

Security ／ セキュリティ

AI駆動開発のセキュリティガイドライン

テンプレート（田中さんと佐藤さんが策定したガイドライン）

AI活用セキュリティガイドライン.md
# AI活用セキュリティガイドライン

  ## 前提: データの取り扱いについて
  - Claude Codeに入力したコードや会話内容は、モデルの学習（トレーニング）には使用されません（Anthropic公式ポリシー）
  - ただし、上記ポリシーに依存せず、以下のルールを守ること

  ## AIに入力してはいけない情報
  - 顧客の個人情報（氏名、メールアドレス、住所等）
  - 認証キー、APIキー、パスワード
  - 本番環境の接続情報
  - 未公開のビジネス戦略情報

  ## 利用ルール
  - Claude Codeの出力は必ず人間がレビューしてからコミット
  - CLAUDE.mdに機密情報を含めない
  - MCPサーバーの追加はチームリーダーの承認を得ること

  ## AI出力の確認フロー
  1. AI生成コードは通常のPRレビューと同じプロセスを経る
  2. セキュリティに関わるコード（認証・認可・暗号化）は必ず2名以上でレビュー
  3. AI生成コードであることをコミットメッセージに明記する
     （例: 「Claude Code支援で実装」）

  ## インシデント対応
  - 機密情報を誤ってAIに入力した場合は、速やかにチームリーダーに報告する
  - 該当するAPIキーやパスワードは即座にローテーションする

ガイドラインのポイントは3つ

1

禁止事項を明確にする

「何を入力してはいけないか」をリスト化することで、チームメンバーが判断に迷う場面を減らす。曖昧なルールは守られない。具体的なリストがあれば、誰もが同じ基準で判断できる。

2

フローを定義する

「AIが出力したコードをどう扱うのか」のフローを決めておくことで、品質とセキュリティの担保が仕組み化される。属人的な判断ではなく、プロセスとして組み込むのがポイント。

3

インシデント対応を想定する

どんなに注意しても、ミスは起こる。重要なのは「ミスが起きたときにどうするか」を事前に決めておくこと。報告のハードルを下げ、対応手順を明確にしておくことで、被害を最小化できる。

Features ／ 特徴

Agent Teams

Agent Teamsは、複数のAIエージェントがチームとして協調作業する機能。Subagent（Task tool）が「1対多の並列実行」だったのに対し、Agent Teamsは「役割を持ったチームの編成と協調」を実現する。

Subagent vs Agent Teams
 Subagent（第5章）Agent Teams
構造親エージェントが子を生成リーダーがチームを編成
役割分担同じ種類のタスクを並列実行異なる役割のエージェントが協調
コミュニケーション親→子の一方向リーダー↔メンバー間の双方向メッセージ
タスク管理親が直接管理共有タスクリストで進捗を管理
適したシーン同種の作業を高速に処理複数の専門性が必要な複合タスク
チームの構成要素
1

リーダーエージェント：タスクの分解、メンバーへの割り当て、全体の整合性チェックを担当

2

メンバーエージェント：それぞれの専門領域で作業を実行。エージェントの種類によって得意分野が異なる

3

タスクリスト：チーム全体で共有される作業リスト。タスク間の依存関係（「タスクAが完了してからタスクBを開始」）も定義可能

4

メッセージング：リーダーとメンバー間、メンバー同士でのコミュニケーション手段

エージェントの種類と使い分け
エージェント種別得意な作業利用できるツール
general-purposeコードの実装・編集全般ファイル操作、Bash、検索など全ツール
Exploreコードベースの調査・分析読み取り・検索系（編集不可）
Plan設計・計画立案読み取り・検索系（編集不可）
Bashコマンド実行に特化Bashコマンドのみ
活用シーンの例
田中さん → Claude Code
# 新機能開発（調査→実装→テストの流れ）
  通知モジュールにメール通知機能を追加して。
  チームを編成して、調査・実装・テストを分担して進めて。

  # フロントエンド + バックエンドの並列開発
  ユーザー設定画面を作って。APIエンドポイントとReactコンポーネントを並列で開発して、最後に結合テストをして。

  # 大規模リファクタリングのモジュール分割
  OrderService,PaymentService,NotificationService のリファクタリングをチームで分担して進めて。
  各モジュールの変更が他に影響しないことを確認して。
注意点
⚠️ 使ううえで押さえておくこと
  • Agent Teamsは現在Research Preview（実験的機能）です。今後、仕様が変更される可能性があります。
  • 複雑なタスクでは精度が落ちることがあります。最終的なレビューは必ず人間が行ってください。
  • 各エージェントの作業結果は独立しているため、結合時の整合性チェックが特に重要です。
  • チームの規模は、タスクの複雑さに応じて2～5体程度が実用的です。
