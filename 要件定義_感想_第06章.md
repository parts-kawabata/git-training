# 第06章「［準備編］企画を確認する」感想文
1. この章の要点（自分の言葉で3行）：ソフトウェアを作るとき、まずは企画を確認する。その企画が不明瞭にならないために、企画書を作成するときは最低限、「プロジェクトの名称」「目的」「目的達成のために作るもの」「作るものの説明」「作るものを利用する人」「利用する人が得られる便益」「作るための体制」「期限」の項目を表記した内容にする。「企画を通すための企画書」ではなく、「『通った企画がいったいどのようなものなのか』を『企画を実現するために作業する関係者』が『理解する』ために必要な企画書」を作成することが重要になる。
2. なるほど／腑に落ちた点（1〜2個）：企画書がないと、なぜこれを作るのか、期限はいつなのかが分からず、要件定義だけでは納得できない。
3. 身近な例で言いかえると（教本の例、または「入金消込」のような“想像でわかる”例でOK。実際の自社業務は知らなくてよい）：旅行に行くときに、「ここに行く」とだけ言われても、どこに集合なのか、いつからいつまでいるのかなど、分からないことばかりで、楽しめる旅行にならないため、日程や誰が参加するのか、どこに行くのかが確定して知らされている方が、安心して旅行の準備ができる。
4. 疑問・もっと知りたい点（1個）：企画に最低限、まとめておくべき内容の1つでも抜けていた場合、どんな変化があるのか
5. 前の章・V字（開発工程）とのつながり（1行）：要件を定めるための事前準備の初めにすること

①目的（なぜ作るか）が無いまま進めると、何が起きる？
- どういうものを作りたいかを考えるとき、目的を決めているとその目的に合ったものを作ることができるが、目的が決まっていなかったら、想像していた完成形から大きく外れたものになる可能性があり、初めから作り直しが必要になってしまう。
