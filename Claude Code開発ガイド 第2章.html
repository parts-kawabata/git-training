Prologue ／ あらすじ

規約は、毎回伝えるしかないのか。

“

自信を持って送り出したプルリクエスト。だが返ってきたのは、賞賛ではなかった。「ここは、チームの規約に沿っていない」——コードのあちこちに、そう告げるレビューコメントが並んでいた。動く。確かに動く。けれど、それは“チームの書き方”ではなかった。

田中はコメントに一つずつ返信しながら、指摘された箇所を手で直し始める。だが、修正すべき場所は一つのファイルには収まらない。複数のファイルにまたがった訂正は、思いのほか時間を溶かしていく。スクロールする指が、ふと止まった。——Claude Codeに、最初からチームのルールを教えておく方法は、ないのだろうか。

答えを探して、田中はまた佐藤さんのもとへ向かった。事情を聞いた佐藤さんは、迷いなく一つの名前を口にする。「それなら、“CLAUDE.md”で解決できるよ」。

教わったとおりに CLAUDE.md を書き、同じような依頼を、もう一度 Claude Code に投げてみる。——すると。レビューで指摘されたはずの箇所は、最初から規約に沿った形で書かれていた。ルールを守ったコードが、一度で、目の前に生成されていた。

一体なぜ、CLAUDE.md を作成しただけで、一度でチームの規約に沿ったコードを生成できたのか。そして——CLAUDE.md とは、どういうものなのか。

What happened ／ 舞台裏

田中さんが CLAUDE.md・Skillsを使って行っていたこと

あの“一度で規約どおり”の裏で、田中さんは2つの仕組みを用意していた。順を追って覗いてみよう。
CLAUDE.md とは ── プロジェクトのルールを書いておくと、Claude Code が起動時に自動で読み込み、最初から規約に沿ってコードを書いてくれる「AIへの申し送り書」。
Skills（カスタムコマンド）とは ── よく使う指示を定型文として登録し、「/review」のような短いコマンドで呼び出せる仕組み。

CLAUDE.md の作成 ─ チームの規約を一枚に書き出す

ファイル：CLAUDE.md
# CLAUDE.md - 工数管理システム（TimeTracker）

## プロジェクト概要
Spring Boot 3.x + Java 17 で構築する社内工数管理ツール。
REST APIを提供し、フロントエンドはReact（別リポジトリ）から呼び出す。

## コーディング規約

### 命名規則
- 変数名・メソッド名: lowerCamelCase
  - 良い例: userId, getOrderCount(), isActive
  - 悪い例: user_id, UserID, get_order_count()
- クラス名: UpperCamelCase
  - 良い例: UserService, OrderRepository
  - 悪い例: userService, user_service
- 定数: UPPER_SNAKE_CASE
  - 良い例: MAX_RETRY_COUNT, DEFAULT_PAGE_SIZE
  - 悪い例: maxRetryCount, max_retry_count

### 例外処理
- 素のRuntimeExceptionは禁止。必ずCustomExceptionを継承した例外を使う
- 理由: 例外の種類によってHTTPステータスコードを自動マッピングするため
  - BusinessLogicException → 400 Bad Request
  - ResourceNotFoundException → 404 Not Found
  - ApplicationException → 500 Internal Server Error

### Repository層
- 戻り値はOptional型で統一する
  - 良い例: Optional<User> findByUserId(String UserId)
  - 悪い例: user findByUserId(String UserId) // nullを返す可能性

### アーキテクチャ
- Controller → Service → Repository の3層構造を守る
- 各層の責務:
  - Controller: リクエストの受付、バリデーション、レスポンス整形
  - Service: ビジネスロジック
  - Repository: データアクセス
- DTOとEntityは必ず分離する。Entityを直接レスポンスに使わない

## 禁止事項
- System.out.println()でのログ出力 → SLF4J（@Slf4j）を使うこと
- マジックナンバーの直書き → 定数クラスに定義すること
- SQLの直書き → Spring Data JPAのRepositoryメソッドを使うこと
- Controllerから直接Repositoryを呼ぶこと → 必ずServiceを経由すること

「AIに伝わるように」と意識して、自分の頭の中になんとなく入っていた知識を、「なぜこのルールがあるのか」「良い例と悪い例はどう違うのか」まで掘り下げて書く必要がある。

🔰 はじめての人へ ─ ここに出てくる言葉
  • CLAUDE.md：プロジェクトのルールを書いた“申し送り書”。Claude Code が起動時に自動で読む。
  • lowerCamelCase / UpperCamelCase / UPPER_SNAKE_CASE：名前の付け方の流儀。変数は小文字始まり、クラスは大文字始まり、定数は全部大文字＋アンダースコア。
  • CustomException：自分たちで定義した例外。種類ごとにHTTPの応答コードを切り替えられる。
  • Optional型：「値が無いかもしれない」を安全に表す入れ物。null の事故を防ぐ。
  • DTO と Entity の分離：DBの形（Entity）と、外に返す形（DTO）を分けて、内部構造を露出させない。
  • SLF4J（@Slf4j）：きちんとしたログ出力の仕組み。println の代わりに使う。

ファイル：CLAUDE.local.md
# CLAUDE.local.md - 田中の個人設定

## 作業スタイル
- コミットメッセージは日本語で書く
- コードにはJavadocコメントを積極的に書く

CLAUDE.md を置いたプロジェクトで、Claude Code に依頼を出す

田中さん → Claude Code
工数入力のエンドポイントを作ってください。ユーザーIDとプロジェクトIDと
作業時間を受け取って、工数レコードを作成するPOST APIです。
🧩 Claude Codeが生成したコードは
  • 変数名は lowerCamelCase
  • リクエストとレスポンスにはDTOが使われている
  • Controller から直接 Repository を呼ばず、Service を経由している
  • Repository の戻り値は Optional 型で、見つからない場合は ResourceNotFoundException（CLAUDE.mdに書いた CustomException の派生クラス）
  • null チェックではなく orElseThrow を使っている
  • 前回のコードと比べると、RuntimeException が使われ、変数名も userID や project_id が混在し、DTOとEntityの分離もされていなかった
🔰 いま起きたこと
  • CLAUDE.md に書いたルールを Claude Code が自動で読み込み、指示していないのに規約どおりのコードを書いた。
  • orElseThrow：値が無いとき、null を返す代わりに例外を投げる安全な書き方。
  • 「前回のコード」とは、CLAUDE.md が無かった第1章の生成物。比べると違いは一目瞭然。

Skills（カスタムコマンド）の作成

1つ目：コードレビューの観点チェック用

ファイル：.claude/commands/review.md
<!-- .clade/commands/review.md -->
以下の観点でコードレビューしてください:

1. コーディング規約の遵守（CLAUDE.mdの規約に沿っているか）
2. エラーハンドリング（例外処理は適切か、CustomExceptionを使っているか）
3. セキュリティ（SQLインジェクション、XSS等の脆弱性がないか）
4. パフォーマンス（N+1問題、不要なDBアクセスがないか）
5. テスタビリティ（テストしやすい設計になっているか）

問題が見つかった場合は、該当箇所のコードと改善案を具体的に提示してください。

対象: $ARGUMENTS

2つ目：テスト生成の定型指示

ファイル：.claude/commands/test.md

以下のクラス/メソッドのユニットテストを作成してください:

## テスト方針
- テストクラス名: 対象クラス名 + Test（例: UserServiceTest）
- テストメソッド名: test + 対象メソッド名 + テストケースの説明（日本語可）
- 正常系テスト: 少なくとも2ケース
- 異常系テスト: 少なくとも2ケース（例外が正しくスローされることを確認）
- 境界値テスト: 該当する場合は1ケース以上

## 使用するライブラリ
- JUnit 5
- Mockito（依存クラスのモック用）
- AssertJ（アサーション用）

対象: $ARGUMENTS

ファイル中の $ARGUMENTS は、コマンド実行時に引数で置き換わるプレースホルダーであり、 /review TimeEntryService.java と打つと、ファイル中の $ARGUMENTS が TimeEntryService.java に置き換わって、Claude Codeへの指示文が完成する。

🔰 ひとこと
  • Skills（カスタムコマンド）：よく使う指示をファイルに保存し、「/review」などの短いコマンドで呼び出せる仕組み。
  • $ARGUMENTS：コマンドの後ろに書いた文字（ファイル名など）が、ここに自動で差し込まれる“穴”。
⚡
田中さんはCLAUDE.mdを書くのに半日かかったが、それは一度きりの投資であり、これからClaude Codeに依頼するたびに、チームの規約に沿ったコードが最初から生成される。Skillsを使うと、Claude Codeがレビュー観点に沿って、コードの問題点を指摘してくれる。変数名のチェック、例外処理の確認、N+1問題の検出など、毎回プロンプトで観点を説明する必要がなくなった。
Column ／ コラム

CLAUDE.md は「人間のため」にもなる

CLAUDE.md は AI のために書く文書だ。——のはずだった。ところが書き上げてみると、いちばん助かったのは“人間”だった。

🧠 暗黙知の言語化

多くのチームでは、コーディング規約やアーキテクチャの方針が「なんとなく共有されている」状態。ベテランメンバーの頭の中にはあるけれど、ドキュメントにはなっていない。CLAUDE.mdを書く作業は、この暗黙知を言語化する強制力を持っている。

「AIに伝わるように書く」という制約が、実はドキュメントの質を上げてくれる。人間向けのドキュメントなら「常識でしょ」と省略してしまう部分も、AI向けには具体例付きで書く必要があり、結果として、人間が読んでも分かりやすいドキュメントができあがる。

🚀 新メンバーのオンボーディング

プロジェクトの技術スタック、アーキテクチャの概要、コーディング規約、禁止事項など、新しく入ったメンバーが最初に知りたい情報がCLAUDE.mdにはコンパクトにまとまっているため、CLAUDE.mdをそのまま新メンバーのオンボーディング資料として使うところもある。

💡
つまり、「AIのために書いた文章」が「チームのため」にもなる。
Mechanism ／ 仕組み

CLAUDE.md の仕組み

配置場所と読み込みの優先順位

CLAUDE.mdは、プロジェクト内の複数の場所に配置することができ、Claude Codeは起動時にこれらを自動的に読み込み、コンテキストとして利用する。

配置場所用途読み込み
~/CLAUDE.mdユーザー全体の共通設定常に読み込まれる
プロジェクトルールの CLAUDE.mdプロジェクト共通の規約プロジェクト内で作業時に読み込まれる
サブディレクトリの CLAUDE.mdモジュール固有の規約そのディレクトリ内のファイルを操作するときに読み込まれる

複数のCLAUDE.mdが存在する場合、それぞれの内容がマージされて適用され、より具体的な（深い階層の）設定が優先されるため、プロジェクト全体のルールをルートに書き、特定モジュール固有のルールをサブディレクトリに書く、という使い分けができる。

チーム共有用とローカル用の使い分け
種類ファイル名Git管理用途
チーム共有CLAUDE.mdするコーディング規約、アーキテクチャ方針、禁止事項
ローカルCLAUDE.local.mdしない（.gitignoreに追加）個人の作業メモ、好みの設定、実験的なルール

チーム共有のCLAUDE.mdをGitで管理することで、チーム全員のClaude Codeが同じルールで動作する。コーディング規約が更新されたら、CLAUDE.mdも一緒に更新してコミットすることで、ルールの変更がチーム全体に即座に反映される。

効果的なCLAUDE.mdの書き方のコツ
1

具体例を必ず添えるー「lowerCamelCaseを使う」だけでなく、良い例・悪い例を併記する

2

禁止事項には理由を書くー「RuntimeException禁止」だけでなく「HTTPステータスコードの自動マッピングのため」と理由を書く

3

プロジェクトの文脈を最初に書くー技術スタック、全体構成、依存関係など、プロジェクトの前提情報があるとAIの出力精度が上がる

4

短く保つーCLAUDE.mdが長すぎると、重要なルールが埋もれるため、A4で1～2ページ程度が目安

Lessons ／ 学び

この章で田中さんが得たもの

1

CLAUDE.mdを書く作業は、チームの暗黙知を言語化する作業でもある

2

チーム全体のルールはCLAUDE.mdに、自分だけの好みはCLAUDE.local.mdに

3

例外処理のルール一つとっても、「CustomExceptionを使う」だけでは不十分で、理由まで書いて初めて、AIは適切な例外クラスを選べるようになる

4

チーム共有のコマンドは .claude/commands/ に置いてGitで管理するため、チーム全員が同じコマンドを使える。個人用のコマンドは ~/.claude/commands/ に置くため、どのプロジェクトでも使える。

5
